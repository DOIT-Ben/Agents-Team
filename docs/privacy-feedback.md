# Feedback Privacy

Agents-Team 不应把反馈做成静默遥测。这个项目处理的是开发仓库，日志、路径、Issue 内容和测试输出经常包含敏感信息，因此反馈必须是 opt-in。

## 默认承诺

| 项 | 承诺 |
| --- | --- |
| 源码 | 默认不收集、不上传 |
| 日志 | 默认不上传，除非用户主动选择片段 |
| 测试输出 | 默认不上传完整输出 |
| 远端 URL | 默认脱敏 |
| 用户身份 | 默认不收集账号、邮箱、token |
| 后台上报 | 不做静默后台上报 |
| Issue 提交 | 默认只生成草稿，用户确认后才创建 Issue |

## 本地导出边界

Agents-Team 的反馈导出是本地能力：默认只打印脱敏 JSON 预览，不写文件、不上传、不创建 GitHub Issue。只有用户显式使用 `--apply` 时才会写入本地脱敏文件。

Agents-Team 的反馈 Issue 提交也是 opt-in：`submit-team-feedback` 和 `scripts/submit_feedback.py` 默认只生成脱敏 Issue 草稿；只有用户审阅草稿并显式确认后，才允许使用 `--apply` 调用 GitHub CLI 创建 Issue。是否提交、提交到哪里、提交哪些片段仍由用户决定。

这意味着本地导出和 Issue 草稿都不是遥测系统，也不是后台收集器。它只是帮用户把准备公开的反馈片段先脱敏、先预览。

## 可收集的低风险元数据

在用户确认后，可以提交这些信息：

- 插件版本。
- 操作系统。
- Codex 使用形态。
- 项目类型和语言栈。
- 当前阶段：安装、初始化、执行、验收、CI、发布。
- 错误码或脱敏错误摘要。
- 用户对流程清晰度、信任度和阻塞点的描述。

## 需要谨慎的内容

这些内容只有在用户主动选择、预览、脱敏后才可以提交：

- dry-run 输出。
- PR 门禁失败摘要。
- 测试命令名称和退出码。
- 文件路径清单。
- 片段级测试日志。

## 禁止自动提交的内容

- `.env`、密钥文件、凭据文件。
- 完整仓库压缩包。
- 完整源码目录。
- 完整 CI artifact。
- 未脱敏的远端 URL、私有 issue、客户信息或个人信息。

## 推荐脱敏规则

| 类型 | 处理方式 |
| --- | --- |
| Windows 用户目录 | `C:\Users\<user>` |
| GitHub 私有仓库 URL | `https://github.com/<owner>/<repo>` |
| 邮箱 | `<email>` |
| token / key / secret | `<redacted-secret>` |
| 客户或项目代号 | `<redacted-name>` |

如果你不确定某段信息能不能公开，默认不要提交。
