# Beta Quickstart

Agents-Team Beta 的目标不是让你立刻改生产项目，而是用一个可回滚的小任务验证：Codex 的长任务能不能从聊天推进，变成 Goal、边界、证据和独立 QA 都可检查的交付流程。

## 适用场景

推荐使用：

- 已经用 Codex 处理跨文件任务或长任务。
- 项目托管在 GitHub，愿意用 Issue 和 PR 记录目标与证据。
- 希望把 Goal、任务边界、测试证据、独立 QA 和反馈闭环固化下来。

暂不推荐：

- 没有 Git 仓库或无法创建分支。
- 不能接受任何项目模板或 CI 文件变更。
- 需要静默后台日志采集、自动上传源码或全自动远程上报。

## 10 分钟试用路径

1. 选择一个测试仓库或可回滚的业务仓库。
2. 安装插件：

```bash
codex plugin marketplace add DOIT-Ben/Agents-Team --ref master
```

3. 重启 Codex，并启用 `agents-team`。
4. 在目标仓库中对 Codex 说：

```text
初始化团队协作机制
```

5. 只检查 dry-run 输出，重点确认：

| 项 | 看什么 |
| --- | --- |
| 技术栈 | 是否识别了正确语言、框架和包管理器 |
| 测试命令 | 是否识别了可运行的最小验证命令 |
| 文件清单 | 是否只新增或修改协作机制相关文件 |
| 冲突项 | 是否保留既有 `AGENTS.md`、Issue 模板、PR 模板和 CI |
| 未确认项 | 是否需要你补充命令、路径或仓库规则 |

6. 如果 dry-run 合理，再明确允许应用到新分支。
7. 创建一个小的 L2 Goal Issue，示例：

```text
Goal: 修复 README 中一个过期命令。
必须完成: README 命令更新；相关文档同步。
验收门禁: 文档链接可读；项目验证命令通过。
任务边界: 不改插件运行逻辑，不改 CI。
风险等级: L2。
依赖与阻塞: 如果测试命令无法识别，先询问。
```

8. 对 Codex 说：

```text
按照 Issue #123 执行团队目标
```

9. PR 出来后，用新上下文验收：

```text
在未参与实现的全新上下文中独立验收 PR #45，只依据 Issue、最终差异和实际证据给出 PASS 或 FAIL。
```

10. 把安装、初始化、执行、QA 中的卡点按 `Beta feedback` Issue 模板提交；如果要附日志片段，先用本地反馈导出预览脱敏结果。

## 成功标准

一次有效试用至少回答这四个问题：

| 问题 | 通过标准 |
| --- | --- |
| 能不能安全初始化 | dry-run 能清楚展示写入计划，不静默覆盖既有文件 |
| 能不能固化目标 | Issue 能表达 Goal、必须完成、验收门禁和任务边界 |
| 能不能约束 Codex | PR 里有证据、边界说明、测试结果、风险记录和当前 commit 绑定 |
| 能不能独立验收 | 非实现上下文能基于 Issue、PR 和证据给 PASS / FAIL |

## 反馈优先级

最有价值的反馈不是“好不好用”，而是：

- 哪一步不敢继续。
- 哪个提示或文档看不懂。
- 哪个文件差点被误改。
- 哪个测试命令识别错了。
- 哪个门禁太重或太轻。
- 哪个失败信息无法指导下一步。
- 哪个反馈片段不知道能不能安全提交。

提交反馈前请阅读 [feedback.md](feedback.md) 和 [privacy-feedback.md](privacy-feedback.md)。
