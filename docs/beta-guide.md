# Agents-Team 邀请制 Beta 指南

Beta 用于验证协作机制是否提高真实交付质量，不以下载量作为成功指标。稳定通道保持关闭，测试者使用固定标签安装：

```powershell
codex plugin marketplace add DOIT-Ben/Agents-Team --ref v0.4.0-beta.1
```

每位测试者获得一个不含姓名或账号的 `cohort_id`，例如 `B1-07`。邀请名册由 Beta 负责人私下保存，记录联系渠道、安装状态和退出状态，不进入本仓库。

## 参与流程

1. 完成安装与一次指定任务。
2. 完成一个自己的真实任务。
3. 至少提交一次本地诊断摘要，包括成功案例也可以。
4. 如需退出，执行适配器移除预览、确认卸载，并自行删除本地运行记录。

## 回访节奏

- **第 1 天**：确认是否安装成功、第一处卡点和环境信息。
- **第 7 天**：收集真实任务结果、误拦截、漏检、耗时和成本感受。
- **第 14 天**：确认是否继续使用、退出原因及诊断材料是否足以复现。

不再使用的测试者也必须回访。负责人每两周汇总邀请数、有效测试者、运行数、反馈数、成功率、漏检率、误拦截率、成本变化、新增回归用例和未解决高风险问题。

仓库已启用 GitHub Discussions，使用 GitHub 默认分类承接四类内容：`Q&A` 对应“使用问答”，`General` 对应“Beta 体验”，`Show and tell` 对应“成功案例”，`Ideas` 对应“机制设计建议”。安全漏洞只使用 Private Security Advisory。

## 反馈运营

反馈按 `new → triaged → needs-information / reproduced → accepted / rejected → regression-added → released → reporter-validated → closed` 流转。公开 Issue 只保存脱敏材料；测试者联系方式和邀请状态保存在维护者私有名册。

- P0 安全、隐私和数据破坏问题当天响应并停止新分发。
- P1 安装、回滚和错误删除问题在两个工作日内响应。
- 每周分类、复现、合并重复反馈，把确认缺陷加入回归数据集并回访已修复问题。
- 每两周发布一次 Beta 汇总，决定继续、扩大、修订或暂停。
- 每月复查数据保留、脱敏规则、Provider 风险和版本采用情况。

仓库已创建 `status:*` 流转标签和七类 `feedback:*` 标签。合并前仍需指定 Beta 负责人，并在维护者私有名册中准备首批邀请对象和联系渠道。
