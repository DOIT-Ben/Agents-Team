# AGENTS.md Guide

`AGENTS.md` 是 Agents-Team 落到项目里的第一块“协作公告牌”。Codex 进入仓库后会先读它，理解这个项目的目标规则、风险边界、测试要求和交付习惯。

一句话：**`AGENTS.md` 让 Codex 知道这个项目应该怎么干活，而不是每次都从聊天里重新猜。**

## 它有什么用

| 场景 | 没有 AGENTS.md | 有 Agents-Team 管理块 |
| --- | --- | --- |
| 开新任务 | 目标、边界、测试靠聊天记忆 | 先看 Goal、风险等级和验收门禁 |
| 跨会话继续 | 下一轮 Codex 重新理解项目 | 规则写在仓库里，新会话也能读到 |
| 防止越界 | “顺手优化”很难拦 | 明确哪些文件、路径、行为需要停下来确认 |
| 验收结果 | 实现者容易自己说完成 | L2/L3 要独立 QA，不能自签 PASS |
| 项目迁移 | 规则散在对话和个人习惯里 | 规则随仓库走，团队成员和 Agent 都能看到 |

## 初始化后会发生什么

当你对 Codex 说：

```text
初始化团队协作机制
```

Agents-Team 会先做 dry-run，展示准备新增或修改的内容。确认后，它会在目标项目中写入一组协作文件，其中最重要的是：

| 文件 | 作用 |
| --- | --- |
| `AGENTS.md` | 给 Codex 看的项目级协作规则 |
| `.codex/team-collaboration.json` | 机制配置、风险策略、测试命令和受保护路径 |
| `.github/ISSUE_TEMPLATE/*.yml` | Goal Issue 表单 |
| `.github/pull_request_template.md` | PR 证据和 QA 记录模板 |
| `.github/workflows/collaboration-gate.yml` | 可选 CI 门禁 |
| `.codex/scripts/*.py` | 本地验证和管理脚本 |

已有 `AGENTS.md` 不会被静默覆盖。Agents-Team 只会维护一个明确标记的管理块，管理块外的项目原有说明必须保留。

## 管理块长什么样

初始化后的 `AGENTS.md` 会包含一个团队协作机制块，核心规则类似：

```md
## 团队协作机制

本项目已启用 `agents-team`。开始开发任务前，读取 `.codex/team-collaboration.json`。

- 所有正式任务必须先有清晰、可观察的 Goal。
- L2/L3 必须使用 GitHub Issue。
- “必须完成”缺一项、验收门禁未通过或任务边界被突破时，不得宣布完成。
- L2 必须独立 QA；L3 实施前必须获得用户确认。
- GitHub Issue 管承诺，Pull Request 管事实，QA 管结论。
```

你可以在管理块外继续写自己的项目规则，例如代码风格、部署习惯、业务术语、目录说明。不要手动改管理块内部；升级和修复时由 Agents-Team 维护。

## 常见玩法

### 玩法 1：给新项目装协作纪律

适合刚开始用 Codex 做中长期开发的仓库。

```text
初始化团队协作机制
```

先看 dry-run。确认它识别的技术栈、测试命令、模板写入计划都合理后，再允许应用。

### 玩法 2：把模糊需求变成 Goal Issue

先把需求写进 GitHub Issue，再让 Codex 执行：

```text
按照 Issue #123 执行团队目标
```

Issue 至少要讲清楚：Goal、必须完成、验收门禁、任务边界、风险等级、依赖与阻塞。

### 玩法 3：让 Codex 自己做，但不能自己验收

实现可以交给 Codex，但 L2/L3 验收要用独立上下文：

```text
独立验收 PR #45
```

独立 QA 只看 Issue、最终差异和实际证据，不相信实现者自己的完成叙述。

### 玩法 4：让机制变成项目巡检工具

你可以定期说：

```text
检查团队协作机制
```

它会检查管理块、配置、模板、脚本、CI 和漂移情况，适合在升级 Codex、迁移仓库、多人协作前跑一遍。

### 玩法 5：安全升级或移除

```text
升级团队协作机制
修复团队协作机制
移除团队协作机制
```

除只读检查外，管理操作默认先预览。写入、升级或移除前都需要用户确认。

## 什么时候不需要它

Agents-Team 不要求所有小改动都走完整流程。

| 任务类型 | 建议 |
| --- | --- |
| 改一个错别字 | 直接改，跑最小检查 |
| 局部可逆的小修 | L1，可由 Codex 自证 |
| 用户可见功能或跨模块改动 | L2，走 Issue、PR 和独立 QA |
| 数据、权限、密钥、真实 Provider、生产环境 | L3，先确认方案和回滚 |

## 怎么判断它有没有生效

初始化后可以运行：

```bash
python .codex/scripts/validate_team_collaboration.py .
```

或者对 Codex 说：

```text
检查团队协作机制
```

健康状态应该能证明：

- `AGENTS.md` 管理块存在。
- `.codex/team-collaboration.json` 存在且有效。
- Issue/PR 模板存在。
- 验证脚本存在。
- 管理文件没有被手动篡改或漂移。

## 安全边界

- 初始化、升级、修复和移除都先预览。
- 不静默覆盖已有 `AGENTS.md`、GitHub 模板、CI 或业务配置。
- 不读取 `.env`、密钥文件或不相关凭据内容。
- L3 任务必须暂停并请求用户确认。
- 没有独立上下文时，不能伪造独立 QA。

## 最佳实践

- 把业务规则写在管理块外，机制规则交给 Agents-Team 维护。
- 每个 L2/L3 Goal 都写清验收门禁，不要只写愿望。
- PR 里记录真实命令、退出码、commitSha 和证据链接。
- 失败时保留失败证据，不要只贴“已修复”。
- Beta 阶段遇到卡点，按 `Beta feedback` Issue 模板提交反馈。
